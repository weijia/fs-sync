# fs-sync 需求文档

> 文档版本：v0.1（草案）　|　最后更新：2026-07-09　|　状态：待评审

## 1. 背景与目标

### 1.1 背景

现代前端应用与跨端应用越来越需要持久化、可同步的文件存储能力：笔记应用、离线文档、配置同步、小型数据库等场景都希望拥有一套"像本地磁盘一样好用"的文件接口，同时数据能在多端之间保持一致。

Node.js 提供了成熟且被广泛使用的 `fs` 模块，但：

- 浏览器没有 `fs`，只有 `IndexedDB` / `OPFS` / `localStorage` 等底层存储；
- 跨端代码若直接使用 `fs`，在浏览器中无法运行；
- 现有同步方案（如各网盘的 SDK）零散、API 不统一，且与本地文件读写解耦，难以做到"读写即同步"。

### 1.2 目标

设计一个名为 **`fs-sync`** 的 TypeScript 库，满足：

1. 对外暴露**与 Node.js `fs` 兼容**的文件操作 API，应用层无需区分运行环境；
2. 在**浏览器与 Node.js** 中都能运行，本地缓存自动适配底层存储；
3. 内置**本地缓存层**，离线可读写文件及目录结构；
4. 支持将本地变更**自动双向同步**到多种远端目标（WebDAV、GitHub、Gitee、RemoteStorage 等）；
5. 支持**冲突检测与解决**，默认"最后写入获胜"，策略可插拔；
6. 提供**统一扩展接口**，第三方可接入任意兼容 `fs` 的同步目标。

### 1.3 非目标（本次范围外）

- 不实现分布式锁 / 强一致事务文件系统；
- 不替代专业数据库（如 IndexedDB 上的关系型查询引擎）；
- 不内置端到端加密（可留作 Provider/拦截器扩展点，但本期不实现）；
- 本次交付**仅设计文档**，不含可运行代码。

## 2. 术语表

| 术语 | 说明 |
| --- | --- |
| Local FS | 库对外暴露的 `fs` 兼容接口 |
| 本地缓存（Local Cache） | 运行环境内的持久化存储（浏览器 IndexedDB / Node 磁盘），保存目录树、文件内容与元数据 |
| SyncProvider | 远端同步目标的统一抽象接口 |
| 同步引擎（Sync Engine） | 负责差异检测、双向同步、冲突解决、重试的协调组件 |
| 元数据（Meta） | 文件描述信息：`path`、`mtime`、`size`、`contentHash`、`version` |
| 冲突（Conflict） | 本地与远端同一路径文件都被修改，且无共同祖先可快进合并 |
| 锚点 / 基线（Baseline） | 上一次成功同步时记录的本地与远端状态快照，用于三方 diff |
| 双向同步 | 本地→远端（push）与远端→本地（pull）同时维护一致 |

## 3. 功能需求

### 3.1 FS 兼容层（FR-1）

| 编号 | 需求 | 说明 |
| --- | --- | --- |
| FR-1.1 | 文件读写 | 实现 `readFile` / `writeFile`（支持 `Buffer`、`string`、编码选项） |
| FR-1.2 | 目录操作 | 实现 `mkdir` / `rmdir` / `readdir` / `rm -rf`（递归删除） |
| FR-1.3 | 文件状态 | 实现 `stat`（含 `mtime`、`size`、`isFile`/`isDirectory`）、`existsSync` |
| FR-1.4 | 重命名/移动 | 实现 `rename`（同卷内移动，含跨目录） |
| FR-1.5 | 删除 | 实现 `unlink`（删除文件） |
| FR-1.6 | 流式 API（建议） | 提供 `createReadStream` / `createWriteStream`，避免大文件内存爆炸 |
| FR-1.7 | 路径归一化 | 统一使用 POSIX 风格路径（`/` 分隔），跨端一致 |
| FR-1.8 | 错误码语义 | 保持 `ENOENT`、`EEXIST`、`EISDIR`、`ENOTDIR` 等错误码与 `fs` 一致 |
| FR-1.9 | 异步优先 | 以 Promise / `async` 风格为主；可附带回调风格包装 |

### 3.2 本地缓存层（FR-2）

| 编号 | 需求 | 说明 |
| --- | --- | --- |
| FR-2.1 | 持久化 | 浏览器落 IndexedDB，Node 落磁盘；进程/刷新后数据仍在 |
| FR-2.2 | 目录树 | 维护完整目录结构，支持 `readdir`、嵌套目录 |
| FR-2.3 | 元数据 | 每个文件记录 `mtime`、`size`、`contentHash`、`version` |
| FR-2.4 | 二进制 | 支持任意二进制内容（`ArrayBuffer` / `Blob` / `Buffer`） |
| FR-2.5 | 离线读写 | 无网络时读写仅命中本地缓存，不阻塞 |
| FR-2.6 | 容量控制 | 浏览器端可配置配额预警与 LRU 淘汰策略（建议） |

### 3.3 同步引擎（FR-3）

| 编号 | 需求 | 说明 |
| --- | --- | --- |
| FR-3.1 | 双向同步 | 同时执行 pull（远端→本地）与 push（本地→远端） |
| FR-3.2 | 增量同步 | 仅同步与基线相比发生变化的文件 |
| FR-3.3 | 自动触发 | 支持文件变更后自动触发、定时器周期触发、手动触发三种模式 |
| FR-3.4 | 差异检测 | 基于元数据（hash / mtime / version）比较本地与远端 |
| FR-3.5 | 重试与退避 | 网络失败指数退避重试，可配置最大次数 |
| FR-3.6 | 幂等性 | 同一变更重复同步不产生副作用或死循环 |
| FR-3.7 | 断点续传 | 同步中断后，依据基线快照与已完成游标续传 |
| FR-3.8 | 多 Provider | 可同时挂载多个远端目标，独立调度 |
| FR-3.9 | 大文件处理 | 大文件分块上传/下载，流式传输，控制内存 |

### 3.4 冲突处理（FR-4）

| 编号 | 需求 | 说明 |
| --- | --- | --- |
| FR-4.1 | 冲突检测 | 本地与远端同一路径均被修改且无法快进合并时识别为冲突 |
| FR-4.2 | 默认策略 | "最后写入获胜"（按 `mtime`/`version`/`hash` 判定） |
| FR-4.3 | 策略可插拔 | 提供 `ConflictResolver` 接口，可自定义 |
| FR-4.4 | 内置策略 | 内置 `LastWriteWins`、`KeepLocal`、`KeepRemote`、`Manual`、`ThreeWayMerge`（建议） |
| FR-4.5 | 事件通知 | 冲突发生时通过事件总线通知调用方 |
| FR-4.6 | 冲突存档 | 冲突时可保留双方版本（如 `.conflict-<side>-<ts>`），便于人工处理 |

### 3.5 Provider 抽象与内置适配器（FR-5）

| 编号 | 需求 | 说明 |
| --- | --- | --- |
| FR-5.1 | 统一接口 | 定义 `SyncProvider`：`authenticate`、`pull`、`push`、`remove`、`stat`、`list` |
| FR-5.2 | WebDAV 适配器 | 内置 WebDAV 实现 |
| FR-5.3 | GitHub 适配器 | 内置 GitHub（基于 Git 仓库/API）实现 |
| FR-5.4 | Gitee 适配器 | 内置 Gitee 实现 |
| FR-5.5 | RemoteStorage 适配器 | 内置 RemoteStorage（remoteStorage.js 规范）实现 |
| FR-5.6 | 自定义扩展 | 第三方通过实现 `SyncProvider` 接入任意兼容 `fs` 的目标 |

### 3.6 可观测性（FR-6）

| 编号 | 需求 | 说明 |
| --- | --- | --- |
| FR-6.1 | 事件总线 | 暴露 `progress`、`conflict`、`error`、`synced` 等事件 |
| FR-6.2 | 状态查询 | 可查询当前同步状态、待同步队列长度 |
| FR-6.3 | 日志 | 可注入 logger，输出同步过程诊断信息 |

## 4. 非功能需求

| 编号 | 类别 | 需求 |
| --- | --- | --- |
| NFR-1 | 跨端 | 同一份 TS 源码在浏览器（ESM）与 Node（CJS/ESM）均可构建运行 |
| NFR-2 | 性能 | 目录树与元数据读取应为 O(1)~O(log n)；大目录分页 |
| NFR-3 | 内存 | 大文件流式，避免整文件入内存 |
| NFR-4 | 健壮性 | 单文件同步失败不影响其他文件；网络抖动自动恢复 |
| NFR-5 | 一致性 | 至少保证最终一致；避免数据丢失（冲突保留双方） |
| NFR-6 | 安全 | 凭据不落明文本地缓存；Provider 鉴权信息隔离 |
| NFR-7 | 可维护 | TypeScript strict；模块边界清晰；接口稳定 |
| NFR-8 | 可测 | 核心逻辑（差异检测、冲突、引擎）可脱离网络单测 |
| NFR-9 | 体积 | 浏览器包尽量小，Provider 可 tree-shaking / 按需引入 |

## 5. 运行环境与适配

| 环境 | 本地缓存后端 | 说明 |
| --- | --- | --- |
| 浏览器 | IndexedDB | 容量大、支持二进制、异步；UI 线程不阻塞 |
| Node.js | 磁盘 `fs` | 直接落盘，路径即真实文件 |

> 其他环境（如 React Native、小程序）可在未来通过实现 `StorageAdapter` 适配，本次仅明确浏览器/Node 两条主线。

## 6. 典型用例

### 用例 1：浏览器中离线写、联网后自动同步

1. 用户打开网页，调用 `fs.writeFile('/notes/a.txt', 'hello')`；
2. 数据写入 IndexedDB，本地可用；
3. 网络恢复，引擎检测到变更并 push 到 WebDAV；
4. 事件 `synced` 触发，UI 显示"已同步"。

### 用例 2：两端同时修改同一文件（冲突）

1. 手机端修改 `a.txt` 并同步；
2. 电脑端离线修改同一 `a.txt`；
3. 电脑联网同步，引擎检测本地与远端均变更且无共同祖先；
4. 触发 `LastWriteWins`，按 mtime 胜出方覆盖；并保留失败方为 `.conflict-remote-<ts>`；
5. 通过 `conflict` 事件通知 UI。

### 用例 3：接入自定义同步目标

1. 开发者实现 `SyncProvider`（如自建对象存储）；
2. 通过 `fsSync.use(provider)` 挂载；
3. 引擎自动将其纳入双向同步调度。

## 7. 边界与约束

- **网络不可用**：所有读写仍可用（命中本地缓存），同步排队延后；
- **凭据缺失**：Provider 未完成 `authenticate` 时，同步报错但不影响本地读写；
- **超大目录**：单次 `list` 必须分页，避免一次拉取上万条目；
- **路径长度**：遵循各运行环境/远端目标的路径长度限制（建议 ≤ 260 字符告警）；
- **循环同步**：本地写 → 触发同步 → 再写本地，必须靠"同步写不打标/去重"避免死循环；
- **删除同步**：`unlink`/`rm` 需在远端同样删除（tombstone），否则下次 pull 会复活。

## 8. 验收标准（建议）

- 给定一组本地变更与远端变更，引擎能在有/无冲突下达成最终一致；
- 浏览器与 Node 下 `readFile`/`writeFile`/`mkdir`/`readdir`/`stat` 行为一致；
- 冲突场景下默认策略正确生效，且双方数据不丢失；
- 网络中断后重试可恢复，且幂等；
- 第三方实现一个 `SyncProvider` 即可被引擎调度。
